# AI Platform — Build Plan

## Why this document exists

QuantumPools is being rebuilt around a unified AI platform: every agent in the product shares one primitive (proposals), one learning loop (corrections flowing back), one event stream (so the product can learn how each org actually works), and one observation surface for the developer (Sonar, the dev-facing intelligence agent).

This plan is **intentionally verbose about "why"** because the developer is one person and context loss between sessions is the single biggest risk to shipping this correctly. A checklist tells you what; it does not tell you why the line item exists, what decision it was responding to, or what breaks if you skip it. Every phase below has a "Why" paragraph stating the decision context, the DNA rule it enforces, and what goes wrong if the phase is skipped or shortcut.

**Remove this doc when all phases are complete.** Until then, update it instead of building from scratch each session.

## Vision (one paragraph)

Every AI output in QP — a suggested job, an estimate draft, an email reply, an equipment match, a workflow config change — flows through the same proposals system. Every user action on a proposal (accept, edit, reject) is a training signal fed back to the responsible agent. Every significant user action in the product emits an event into an observation stream. A workflow-observer agent watches those events per-org and proposes configuration changes to that org's admin. A separate agent (Sonar) watches across orgs (and the product itself) and proposes product changes to the developer. The end state is a product that compounds in usefulness every week it's used — for its users and for its builder.

## DNA rules this plan enforces

Every phase in this plan exists because of one or more of these rules (in `CLAUDE.md` Product DNA section):

1. Build for the 1,000th customer, not Sapphire.
2. Every AI agent learns — no static agents.
3. The product learns the org, not just the agents.
4. Never optimize for ease of build.
5. AI never commits to the customer.
6. Less work for the USER, not the engineer.

Where a phase leans hard on a specific rule, it's called out in that phase's "Why."

## Glossary

| Term | Definition |
|---|---|
| **Event** | A timestamped record of something the user, the system, or an agent did. Written to `platform_events`. Never mutated. |
| **Proposal** | A staged database record representing something an AI wants the user to create/change. Has a `status` lifecycle: `staged` → `accepted`/`edited`/`rejected`/`expired`/`superseded`. Written to `agent_proposals`. |
| **Entity proposal** | A proposal whose target is a domain record (job, estimate, invoice, equipment, etc.). `entity_type` field identifies which. |
| **Meta-proposal** | A proposal whose target is a configuration change on an org (defaults, post-creation handler, rules). `entity_type = "org_config"`. |
| **Product proposal** | A proposal whose target is a product-level recommendation to the developer. `entity_type = "product_recommendation"`. Generated by Sonar. |
| **Agent** | Named AI output surface with a defined `agent_type` string in `AgentLearningService`. Existing: `email_classifier`, `email_drafter`, `deepblue_responder`, `command_executor`, `job_evaluator`, `estimate_generator`, `customer_matcher`, `equipment_resolver`. New (from this plan): `inbox_summarizer`, `workflow_observer`, `sonar`. |
| **Correction** | A structured record of the human's response to an agent's output. Already exists in `agent_corrections`. This plan formalizes that every proposal resolution also produces a correction row. |
| **Handler** | An org-configurable function invoked after a domain event. This plan introduces `post_creation_handler` (per-org) for what happens after a job is created from a proposal. Extensible to other events later. |
| **Sonar** | Dev-facing intelligence agent for the developer (Brian). Reads cross-org + product + code data, surfaces product recommendations. Four surfaces: dashboard, chat, alerts, weekly digest. |
| **User-Sonar** | Future product feature (Phase 8). Same analytics pattern, scoped to a single org, surfaced to that org's admin. Not built in this plan. |

## Current state (as of 2026-04-18)

**What exists:**
- `AgentLearningService` with per-agent corrections in `agent_corrections` table.
- Eight existing agent types listed above.
- DeepBlue tool-use engine with ad-hoc `requires_confirmation` pattern for some tools. Each confirmation renders through a bespoke tool card in `tool-cards.tsx`.
- Ad-hoc instrumentation: some events are logged to application logs or to specific model tables, but there is **no unified event stream**.
- Inbox displays threads with columns (sender, subject, category, status, msg count). No AI summary surface. No proposal surface.
- Email/estimate/invoice sends go through `EmailService` (see `docs/email-pipeline.md`).
- `agent_corrections` only captures explicit corrections — acceptance as a signal is sometimes recorded, often not.

**What doesn't exist:**
- Unified event stream.
- Unified proposal table and lifecycle.
- Post-creation handler abstraction (job creation is hardcoded, not org-configurable).
- Workflow-observer agent.
- Sonar.
- Inbox AI summary.
- Measurement of efficiency-over-time per org (no way to answer "is this org getting faster?").

## Phase 0: Decisions & taxonomy (no code)

### Why this phase exists

The single biggest reason these systems fail in practice is inconsistent event taxonomy. If team A logs `job.created` and team B logs `JOB_CREATED_EVENT`, the analytics layer becomes a janitor forever. Naming, levels, and payload shape are decided once, before a single event is emitted.

**DNA alignment**: rule 4 (never optimize for ease) — it's faster to skip this phase and start logging. It's also irrecoverable; by the time you realize taxonomy is broken, you have months of data in the wrong shape.

### Decisions to lock

1. **Event type naming**: `noun.verb` lowercase dotted, e.g., `job.created`, `proposal.accepted`, `thread.summarized`, `compose.send_clicked`. Never `CamelCase`, never `VERB_NOUN`.
2. **Event payload shape**: every event has `{type, org_id, actor_id|null, entity_refs, payload, created_at}`. `entity_refs` is a map like `{thread_id: "...", customer_id: "..."}` — polymorphic without schema pain.
3. **Event levels**: `user_action` (explicit user did this), `system_action` (backend did this without user input), `agent_action` (AI did this), `error`. Rolling up by level lets us answer "what did the user actually do" separately from "what did the system do on their behalf."
4. **Privacy**: events are org-scoped; cross-org reads require platform-admin role. No PII in payloads — reference by ID only.
5. **Retention**: events are permanent for dogfood/dev orgs, retention-policy-configurable at scale. Retention is a v2 concern; default forever.

### Artifacts produced

- `docs/event-taxonomy.md` — the canonical list of event types and payload schemas. New features add their events to this doc in the same PR.

### Acceptance criteria

- Event taxonomy doc exists.
- Every team member (today: just Brian + Claude) reads it before instrumenting anything.

## Phase 1: Event instrumentation foundation

### Why this phase exists

Nothing else in this plan works without the event stream. Workflow-observer has nothing to observe. Sonar has nothing to analyze. `feedback_product_learns_the_org` is impossible. Even basic debugging ("how did this customer get into this state?") is impossible.

Comprehensive capture from day 1 — because **storing is cheap, backfilling is impossible**. Any event we don't capture today is gone. Brian's explicit concern (2026-04-18): pinned-for-later features die on the vine. Instrumentation is the one thing we cannot defer.

**DNA alignment**: rule 3 (product learns the org) — you cannot learn what you didn't measure. Rule 4 (never optimize for ease) — comprehensive beats narrow.

### Scope

**Backend:**

- New table `platform_events`:
  ```
  id (uuid, pk)
  organization_id (fk, indexed, nullable for platform-level events)
  actor_user_id (fk, nullable for system/agent actions)
  actor_type (user | system | agent)
  actor_agent_type (string, nullable — e.g., "email_drafter")
  event_type (string, indexed — e.g., "thread.summarized")
  level (user_action | system_action | agent_action | error)
  entity_refs (jsonb — polymorphic entity pointers)
  payload (jsonb — event-specific details)
  request_id (string, nullable — trace across events in one request)
  created_at (timestamptz, indexed)
  ```
  Indexes: `(organization_id, created_at desc)`, `(event_type, created_at desc)`, GIN on `entity_refs`.

- New service `src/services/events/platform_event_service.py`:
  ```python
  async def emit(
      event_type: str,
      org_id: str | None,
      actor: Actor,
      entity_refs: dict = {},
      payload: dict = {},
      level: str = "user_action",
      request_id: str | None = None,
  ) -> None
  ```
  - Non-blocking: wraps in try/except, never breaks the primary operation.
  - Writes to DB asynchronously (fire-and-forget task or bulk buffer).

- Middleware: every authenticated request gets a `request_id` attached to `request.state` and logged with every event emitted in that request.

- Instrument **every existing surface** — not "start narrow." Use a codebase-wide audit to identify every mutation and add emit calls. Specifically:
  - All agent output paths (every generate/draft/classify call)
  - All state transitions (job.status_change, thread.status_change, invoice.status_change, visit.completed, etc.)
  - All customer-facing send events (email.send, estimate.sent, invoice.sent, etc.)
  - All user interactions the backend sees (compose.send_clicked, archive.clicked, etc.)
  - All errors (5xx, AI call failures, Postmark failures, etc.)

- For user-side interactions that happen purely frontend (button clicks, page views, hover-expands), add a `/v1/events` endpoint that the frontend posts to. Batch events frontend-side to reduce network chatter (flush every 5s or on 20-event buffer).

**Frontend:**

- Event batching client in `lib/events.ts`:
  ```ts
  emit(type: string, refs?: object, payload?: object)
  // buffers locally, flushes on timer or on route change
  ```
- Instrument page views, inbox row clicks, compose actions, pill clicks, archive/snooze/assign, dashboard interactions.

**Event taxonomy (initial set — expand in Phase 0 doc):**

- `page.viewed` — path, referrer
- `thread.opened` — thread_id
- `thread.archived`, `thread.snoozed`, `thread.assigned` — with prior/new values
- `compose.opened`, `compose.send_clicked`, `compose.discarded`
- `proposal.staged`, `proposal.accepted`, `proposal.edited`, `proposal.rejected`
- `agent.generated` — agent_type, confidence, tokens
- `job.created`, `job.status_change`, `job.assigned`
- `customer.created`, `customer.edited`
- `error.backend_5xx`, `error.ai_call_failed`

### Acceptance criteria

- `platform_events` table exists, migration applied.
- Every existing backend mutation emits at least one event.
- Frontend emits events for every named interaction (button click list in taxonomy doc).
- A query like `SELECT count(*) FROM platform_events WHERE organization_id = '...' GROUP BY event_type` returns useful data after a day of normal usage.
- Event emission failures are logged but do NOT break the primary request (verified via test case).

### What NOT to do in this phase

- Do NOT start narrow and expand. Instrument comprehensively or the later phases are compromised.
- Do NOT build any UI for events yet. This phase is pure data capture.
- Do NOT put PII in payloads. Reference by ID; the target tables have the PII.

### Estimated scope

~1.5 weeks of backend + ~3 days of frontend instrumentation + ~1 day of taxonomy docs.

## Phase 2: `agent_proposals` system

### Why this phase exists

Today, AI suggestions are ephemeral UI state (inbox draft reply) or inconsistent domain-specific records (DeepBlue `requires_confirmation` tool cards). This produces three problems:

1. **Learning is spotty.** When the user edits an AI draft before sending, sometimes we record the correction, sometimes we don't. Depends on the specific code path remembering to call `record_correction`. This is exactly the kind of discipline that degrades over time when a one-person team is busy.
2. **Session durability is broken.** Ephemeral UI state means walking away from a batch of 3 AI-proposed jobs loses them. User has to redo the work. Violates rule 6 (less work).
3. **No unified review surface.** Today, you can't answer "what has AI proposed this week that users didn't accept?" because the data is scattered across 5 tables and 3 memory-only UI states.

A first-class proposals system fixes all three simultaneously: learning is enforced at the data layer; proposals survive sessions; one surface answers "what's the AI doing." And the same architecture handles entity proposals (jobs, estimates), meta-proposals (org config), and product proposals (Sonar's output).

**DNA alignment**: rule 2 (every agent learns — enforced structurally), rule 4 (never optimize for ease — replacing one-off modals with a generic pattern is more work), rule 5 (AI never commits — the proposal IS the drafted thing awaiting approval).

### Scope

**Backend:**

- New table `agent_proposals`:
  ```
  id (uuid, pk)
  organization_id (fk, indexed)
  agent_type (string — e.g., "inbox_summarizer", "workflow_observer", "sonar")
  entity_type (string — "job", "estimate", "org_config", "product_recommendation", etc.)
  source_type (string — "agent_thread", "agent_message", "visit", "observation_batch")
  source_id (string, indexed — references source_type's table)
  proposed_payload (jsonb — the fields that would commit to the target entity)
  confidence (float — AI's self-reported 0-1)
  status (string — staged | accepted | edited | rejected | expired | superseded)
  rejected_permanently (boolean — "never propose this pattern again")
  superseded_by_id (uuid, fk self, nullable)
  outcome_entity_id (string, nullable — FK to the created record on acceptance)
  outcome_entity_type (string, nullable)
  user_delta (jsonb, nullable — diff between proposed_payload and what user actually committed)
  resolved_at (timestamptz, nullable)
  resolved_by_user_id (fk, nullable)
  resolution_note (text, nullable — user's optional reason for reject)
  created_at, updated_at
  ```
  Indexes: `(organization_id, status, created_at desc)`, `(source_type, source_id)`, `(agent_type, status)`.

- New service `src/services/proposals/proposal_service.py`:
  - `stage(agent_type, entity_type, source, payload, confidence, org_id) -> Proposal`
  - `accept(proposal_id, user_id) -> created_entity` — atomically: create target record, mark proposal `accepted`, emit `proposal.accepted` event, trigger learning signal (non-blocking), invoke post-creation handler.
  - `edit_and_accept(proposal_id, user_id, edited_payload) -> created_entity` — same as accept but computes `user_delta`, marks `edited`, records stronger learning signal with the delta.
  - `reject(proposal_id, user_id, permanently=False, note=None)` — mark rejected; if `permanently=True` set `rejected_permanently`; emit event; learning signal.
  - `expire_stale(age_days=30)` — background job that marks unresolved stale proposals as `expired`. Learning signal.
  - `supersede(old_id, new_proposal)` — when AI re-proposes on fresher context.

- Proposal → learning signal bridge: every `accept`/`edit`/`reject` atomically writes a corresponding `agent_corrections` row (so existing learning path keeps working). This is the structural enforcement of rule 2.

- **Entity-type registry** — a pluggable mapping of `entity_type` string → a creator function. Example:
  ```python
  ENTITY_CREATORS = {
      "job": lambda payload, org_id, user_id: AgentActionService.create_action(...),
      "estimate": lambda payload, org_id, user_id: InvoiceService.create(..., document_type="estimate"),
      "org_config": lambda payload, org_id, user_id: OrgConfigService.apply(...),
      # etc.
  }
  ```
  Single exit point for "how proposals become real records." Adding a new entity_type means registering a creator and schema-validating the payload.

- **DeepBlue migration**: migrate every existing `requires_confirmation` tool use in `src/services/deepblue/tools_*.py` to stage a proposal instead of returning an inline confirmation payload. The tool card UI consumes proposals instead of ad-hoc `requires_confirmation` objects. **This reduces net code**, doesn't add to it — removes the bespoke confirmation payload pattern.

**Frontend:**

- New component `ProposalCard` — reusable, rendering shape driven by `entity_type`:
  - Header: agent_type badge, confidence pill, source link
  - Body: type-specific rendering (job fields, estimate fields, org_config diff, etc.)
  - Actions: `Accept`, `Edit & Accept`, `Reject` (with `Never again` checkbox), `Open in full editor`
- Migration: existing DeepBlue confirmation cards re-implemented on top of `ProposalCard`.

**Events to add:**
- `proposal.staged`, `proposal.accepted`, `proposal.edited`, `proposal.rejected`, `proposal.rejected_permanently`, `proposal.expired`, `proposal.superseded`

### Acceptance criteria

- `agent_proposals` table exists.
- `ProposalService` can stage/accept/edit/reject a proposal of at least three entity_types (job, estimate, equipment_item).
- All existing DeepBlue `requires_confirmation` paths migrated to proposals.
- Every resolve operation atomically produces an `agent_corrections` row.
- Every resolve operation emits a `proposal.*` event.

### What NOT to do in this phase

- Do NOT keep any one-off confirmation modals. If a surface skipped migration, it becomes tech debt the next engineer has to hunt down. Migrate it all.
- Do NOT hand-wire per-agent UI cards. Use the generic `ProposalCard` with entity_type dispatch.

### Estimated scope

~1 week backend, ~1 week frontend (includes DeepBlue migration).

## Phase 3: Inbox summarizer agent (first real proposal producer)

### Why this phase exists

Two reasons:

1. **The inbox redesign is what started this whole conversation.** Brian's hypothesis (2026-04-18): most threads can be triaged from the list view if we replace subject/msg-count columns with a summary + staged actions. This phase ships that redesign.
2. **Validates the architecture end-to-end.** Phase 2 built a generic system. Phase 3 is the first non-trivial agent consumer. If the abstraction is wrong, we find out here before workflow-observer and Sonar depend on it.

**DNA alignment**: rule 6 (less work for the user — 1-click triage vs. 5-click triage), rule 2 (agent learns from every summary + proposed action outcome).

### Scope

**Backend:**

- New agent `inbox_summarizer` in `src/services/agents/inbox_summarizer.py`:
  - Input: `AgentThread` + all its `AgentMessage`s + customer context + related cases/jobs/invoices.
  - Output: structured summary fields (`ask`, `state`, `open_items[]`, `red_flags[]`, `linked_refs[]`) + zero or more staged proposals (`entity_type="job"`, `entity_type="estimate"`, `entity_type="case_link"`, etc.).
  - Claude Haiku for cost/speed. Sonnet only if Haiku quality is insufficient (measure first).
  - Uses `AgentLearningService.build_lessons_prompt` to inject prior corrections.
- Storage: summary fields cached on `agent_threads` table (new columns: `ai_summary_payload jsonb`, `ai_summary_generated_at`, `ai_summary_version int`).
- Trigger: regenerate summary on any new inbound message in the thread. Debounce so rapid reply bursts don't regenerate N times.
- Background job: regenerate stale summaries (e.g., summary older than 7 days but thread not archived) to pick up new learning.
- Proposals staged by the summarizer target `agent_thread` as `source_id`.

**Frontend (inbox redesign):**

- Replace current column-based inbox row layout with a two-column card:
  - Left (~45%): From + subject + badges + time/count
  - Right (~55%): summary card (ask / state / open items / red flag) + inline staged proposals (as `ProposalCard` minis) + inline actions (`Accept All`, `Draft Reply`, `Archive`)
- Row height: tall default; compact-mode toggle in inbox settings for power-scan use cases.
- Reading pane unchanged — still available for deep reads, attachments, forwards.
- Short threads (1 msg, nothing AI-worth-summarizing): right column shows snippet, not a stub summary. Decided by confidence threshold + message count.

**Events:**
- `thread.summarized` — summary generated (success or empty)
- `inbox.row.expanded`, `inbox.proposal.accepted` etc. (or lean on the generic proposal events)
- `inbox.compact_mode_toggled`

### Acceptance criteria

- Summaries generate for every inbound-message-arriving thread within ~30s.
- Stale summaries regenerate on a schedule.
- Short threads handled gracefully (no stub summaries).
- Inbox redesign ships behind a per-org feature flag — default off, enable for Sapphire first.
- After 1 week of Sapphire use: measure time-from-thread-arrival-to-triage-decision vs. old inbox. Expect 30-50% reduction.

### What NOT to do in this phase

- Do NOT kill the column layout for all orgs immediately. Feature-flag it. Measure.
- Do NOT design the fields for Sapphire's specific patterns. The summary shape should work for a residential-only customer base, a commercial-heavy base, a dispatch-style org, etc.

### Estimated scope

~1 week backend, ~2 weeks frontend + visual design iteration.

## Phase 4: Post-creation handlers (per-org configurable)

### Why this phase exists

When a proposal accepts and creates (say) a job, the user's real workflow continues — the job needs to be scheduled, assigned, routed. Different orgs handle this differently. Hardcoding one pattern violates rule 1 (build for the 1,000th customer). This phase introduces the pluggable handler abstraction.

Secondary: makes workflow-observer (Phase 6) possible — the handler becomes the observable surface ("user opens scheduler 0/47 times → propose different handler").

**DNA alignment**: rule 1 (configurable per org), rule 3 (observable, learnable), rule 6 (less work — the right handler auto-appears based on what the org actually does).

### Scope

**Backend:**

- New table `org_workflow_config`:
  ```
  organization_id (pk, fk)
  post_creation_handlers (jsonb — map of entity_type → handler_name)
    e.g., { "job": "assign_inline", "estimate": "navigate_to_detail" }
  default_assignee_strategy (jsonb — config for assign_inline, etc.)
  updated_at, updated_by
  ```

- New registry `src/services/workflow/handlers.py`:
  ```python
  HANDLERS = {
      "assign_inline": AssignInlineHandler,            # shows assignee picker in card
      "schedule_inline": ScheduleInlineHandler,        # shows date+assignee picker
      "unassigned_pool": UnassignedPoolHandler,        # toast-only, dropped in queue
      "auto_assign_proximity": AutoAssignProximity,    # OR-Tools assignment
      "coordinator_notification": CoordinatorNotify,   # notify a named user
  }
  ```
  Each handler exposes: `ui_spec()` (how the frontend renders the post-creation step) + `apply(user_input)` (the server-side action).

- Default handler selection at org creation:
  - Start with `assign_inline` (sensible generic default).
  - After 2-3 weeks of observation, workflow-observer may propose a different default.

- Expose config in Settings → Workflows (plain-language, opinionated — see below).

**Frontend:**

- New settings surface: **How new jobs get handled**
  - Plain language options with descriptions and use-case hints:
    - "Schedule right away — best for coordinator-driven teams"
    - "Create and assign — best for flexible schedules like ours"
    - "Send to unassigned pool — best for dispatch-style teams"
    - "Auto-assign by proximity — best for large teams with optimized routes"
  - "Based on your activity, [X] is recommended" callout if observation data exists.
  - NO enum dropdowns. NO engineer vocabulary. NO "advanced" panel (if something needs to be hidden behind "advanced," design it differently).

- Inbox proposal cards render the handler's `ui_spec()` inline when the user clicks Accept on a staged job.

**Events:**
- `workflow_config.changed`, `handler.applied`, `handler.abandoned` (user started a handler flow and closed it unused)

### Acceptance criteria

- Sapphire's handler configured to `assign_inline` with default assignee observation (starts null, auto-learns to Kim).
- At least 3 handlers implemented (`assign_inline`, `unassigned_pool`, `schedule_inline`).
- Settings surface exists and is intuitively usable without docs (verify by having Brian configure it without reading the code).
- Workflow-observer (Phase 6) will be able to propose changes to this table.

### What NOT to do in this phase

- Do NOT expose internal handler names (`assign_inline`) in the settings UI. Plain language only.
- Do NOT add a "custom handler" option. If an org needs something we don't have, that's a product signal — we build the handler, not let the user write code.

### Estimated scope

~1 week backend, ~1 week frontend.

## Phase 5: Unify existing agents into the proposal system

### Why this phase exists

Phases 2-4 built the platform. Phases 5+ systematically migrate existing agents to use it. This is the cleanup phase — without it, we have two styles of agent output (old and new) forever, which is exactly the kind of half-finished architecture rule 4 (never optimize for ease) prohibits.

Existing agents to migrate (from `agent-learning-system.md`):

| Agent | Current shape | Migrate to |
|---|---|---|
| `email_classifier` | Auto-applies fields to AgentThread | Stays as classification (not a proposal). Keep existing correction flow. |
| `email_drafter` | Writes `draft_response` on AgentMessage | **Becomes a proposal** with `entity_type="email_reply"`. Draft awaiting approval == staged proposal. |
| `deepblue_responder` | Chat replies (no commit) | No change — conversational, not a proposal. |
| `command_executor` | DeepBlue tool_use with `requires_confirmation` | **Migrated in Phase 2.** Each tool call that mutates state becomes a proposal. |
| `job_evaluator` | Sets fields on AgentAction | Keep as classification. |
| `estimate_generator` | Creates draft estimates via EstimateWorkflowService | **Becomes a proposal** with `entity_type="estimate"`. User accepts the draft = proposal accept. |
| `customer_matcher` | Auto-matches inbound emails to customers | Keep as classification, but add low-confidence match → **proposal** with `entity_type="customer_match_suggestion"` that the user accepts before applying. |
| `equipment_resolver` | DeepBlue tool output | Covered by Phase 2 migration. |

### Scope per agent

For each migrated agent:
- Change the output path to `ProposalService.stage(...)` instead of direct DB write.
- Update the consumer UI (inbox reading pane for email_drafter, estimate detail for estimate_generator, customer-match review queue for customer_matcher) to consume `ProposalCard`.
- Delete the previous ad-hoc confirmation code.
- Verify learning signal is equivalent or stronger than before.

### Acceptance criteria

- No agent in the system produces AI output that requires human accept/edit/reject through a non-proposal path.
- DeepBlue tool cards, email draft-approval timeline, estimate draft banner all render through `ProposalCard`.
- `grep -r "requires_confirmation" src/` returns zero backend results after the migration (audit test).

### Estimated scope

~2 weeks (one agent per 2-3 days depending on surface complexity).

## Phase 6: `workflow_observer` agent

### Why this phase exists

This phase is where **the product learns the org**. Without it, we have data capture (Phase 1) and proposal infrastructure (Phase 2), but nothing that closes the loop from observation → meta-proposal. Rule 3 becomes real code here.

**DNA alignment**: rule 3 explicitly. Rule 2 (this is itself an agent that learns from which of its meta-proposals admins accept vs. reject). Rule 6 (compounds over time — org admin configures less as observer learns more).

### Scope

**Backend:**

- New agent `workflow_observer` in `src/services/agents/workflow_observer.py`.
- Runs as a daily background job per org.
- Reads from `platform_events` + `agent_proposals` outcomes.
- Pattern detectors (pluggable — each is a function that takes events+proposals, returns zero or more meta-proposals):
  - **Default-assignee detector**: ≥80% of job-proposal accepts go to the same user? → propose `default_assignee = that_user` for that entity type.
  - **Handler-mismatch detector**: user opens handler UI and abandons it >70% of the time → propose switching to a handler that matches the abandoned-path behavior.
  - **Classification-override detector**: user consistently changes category X to category Y → propose updating the default or adding a rule.
  - **Time-of-day detector**: >80% of triage happens in a specific window → propose Focus Mode or batching UI.
  - **Rejection-cluster detector**: same `rejected_permanently` pattern across multiple threads → the AI is making a consistent mistake; propose a learning adjustment or rule.
- Output: meta-proposals (`entity_type="org_config"`) staged for that org's owner.
- Threshold: only propose when confidence high enough to avoid noise. Start conservative (>90%); adjust based on acceptance rate of observer's own proposals.

**Frontend:**

- Dashboard widget: "Workflow Suggestions" card showing unresolved meta-proposals from workflow_observer.
- Each meta-proposal: one-sentence explanation, the observed behavior it's based on, "Apply" / "Not now" / "Never suggest this" buttons.
- Settings → Workflows page shows both current config AND pending observer suggestions relevant to each setting.
- Weekly digest email to owner: "This week workflow_observer suggested 3 changes. 2 still pending review."

**Events:**
- `observer.scan_complete`, `meta_proposal.staged`, plus the generic proposal events.

### Acceptance criteria

- After 2 weeks of Sapphire usage, workflow_observer has staged at least one meta-proposal that feels obviously right to Brian.
- No false positives (meta-proposals that Brian rejects as nonsense).
- Rejection rate measured weekly; if >30%, tune thresholds or detector logic.

### What NOT to do in this phase

- Do NOT auto-apply meta-proposals. Every org-config change requires admin accept. Rule 5's spirit applied beyond customer-facing actions.
- Do NOT propose aggressively. Better to miss a valid suggestion than to spam the admin with marginal ones.

### Estimated scope

~2 weeks backend (mostly pattern-detector logic), ~3-5 days frontend surfaces.

## Phase 7: Sonar (dev-facing intelligence)

### Why this phase exists

This is the phase that (per Brian's stated belief, 2026-04-18) determines whether QP succeeds as a product. A one-person developer cannot watch all the data themselves. Without Sonar, user pain points stay invisible, agent regressions go uncaught, feature health is a guessing game, and the product stops improving at the rate it could.

**DNA alignment**: rule 3 applied to the product itself (the app learns about the developer's work). Rule 2 (Sonar is itself a learning agent — learns which recommendation types Brian finds useful). Implicit: this is the moat. Competitors can clone features. They cannot clone a product that knows its own weaknesses and tells its builder what to fix next.

### Scope

**Backend:**

- New agent `sonar` in `src/services/agents/sonar.py`.
- Platform-admin access only. Cross-org data reads gated by role check.
- Tool-use engine (same pattern as DeepBlue) with Sonar-specific tools:
  - `query_event_trends` — time-series aggregation
  - `query_proposal_outcomes` — acceptance/edit/rejection rates by agent, entity, org, customer
  - `query_learning_curves` — is agent X improving over time for org Y?
  - `query_friction_hotspots` — where users abandon/backtrack/repeat
  - `query_feature_usage` — click-through, dwell time, reach
  - `query_error_clusters` — grouped errors with frequency
  - `query_feedback_items` — correlate FB-XX with usage patterns
  - `query_git_activity` — what was changed recently (useful context for "why did X regress")
- Analysis outputs go through `ProposalService.stage(agent_type="sonar", entity_type="product_recommendation", ...)`. Payload shape: `{title, body, evidence_refs, suggested_actions, category}`.
- Runs proactively (daily scan) AND on-demand (Brian asks Sonar a question).
- Learning: Brian's accept/defer/reject on each product_recommendation feeds AgentLearningService with `agent_type="sonar"`.

**Frontend — Sonar dashboard (`/admin/sonar`):**

Single integrated page with four reinforcing modes:

1. **Ambient dashboard (main view)**:
   - Top 5 staged product_recommendations, ranked by Sonar's priority score
   - Trend sparklines for key metrics (agent acceptance rates, time-to-triage, error rate, proposals/day)
   - "What changed since yesterday" strip
   - Recent acceptances/rejections/defers

2. **Conversational drill-in (right-side panel)**:
   - Chat interface. "Why did estimate acceptance drop?" → Sonar queries, synthesizes, renders evidence.
   - Conversation history per user.
   - Links any insight back to a staged recommendation.

3. **Proactive alerts (stream + notification)**:
   - Threshold rules (configurable by Brian): "alert me when [metric] changes by X%"
   - Default rules: acceptance-rate drop, error spike, new rejection cluster, latency regression.
   - Delivered via ntfy (per `ntfy-alerts.md`) plus inbox-style list on the dashboard.

4. **Weekly reflection email**:
   - Summary of the week's data (what improved, what regressed).
   - Links into the dashboard for specifics.
   - Not a replacement for the dashboard — a forcing function to look back.

**Events:**
- `sonar.scan_complete`, `sonar.recommendation_staged`, generic proposal events.

### Acceptance criteria

- Dashboard exists at `/admin/sonar`, platform-admin-gated.
- Sonar generates at least 5 product recommendations in its first week running against dogfood data, and Brian finds ≥3 of them useful.
- Drill-in chat answers at least 10 real "why" questions Brian asks without hallucination.
- Alert rules fire correctly (verified by seeded test event that crosses a threshold).
- Weekly email delivered.

### What NOT to do in this phase

- Do NOT build User-Sonar in this phase (that's Phase 8, future). Dev-Sonar has platform-admin access and cross-org data; User-Sonar is a different product for org admins.
- Do NOT skip the learning loop. Sonar must track which recommendations Brian acts on. Without it, rule 2 is violated and Sonar degrades into noise over time.
- Do NOT surface recommendations without evidence. Every recommendation shows the underlying data query and the events/proposals that support it. If Brian can't verify the claim, the recommendation is worthless.

### Estimated scope

~2 weeks for tool implementations + backend agent, ~2-3 weeks for frontend dashboard (the dashboard is substantial).

## Success criteria for the full plan

After all phases ship, the following should be true:

- Every AI output in the product flows through `agent_proposals`. Zero one-off confirmation modals.
- Every resolution of a proposal produces a learning signal without developer effort.
- Every significant user interaction is in `platform_events`.
- `workflow_observer` has made at least 5 meta-proposals that Sapphire's admin accepted.
- Sonar has made at least 10 product recommendations that Brian acted on, and ≥3 of those measurably improved a metric (acceptance rate, time-to-triage, error rate).
- A new feature can be instrumented, learned-from, and analyzed without the developer writing bespoke analytics code — the platform is the analytics.
- Brian's time spent manually debugging product-level questions drops by at least 50%.
- The product gets measurably faster to use for Sapphire over 6 months (via time-on-task metrics).
